Contexto CTI

Quem é o Salt Typhoon?

⬡ Threat Actor Intelligence

Salt Typhoon (também rastreado como GhostEmperor, FamousSparrow e UNC2286) é um grupo APT vinculado ao Ministério de Segurança do Estado (MSS) da China. Ativo desde pelo menos 2019, opera com foco em espionagem de longo prazo contra telecomunicações, governo e infraestrutura crítica. Em 2024, o CISA emitiu advisory confirmando comprometimento de múltiplas operadoras de telecomunicações norte-americanas (AT&T, Verizon, T-Mobile), com acesso mantido por meses sem detecção.

Características Operacionais
  • Preferência por Living Off the Land (LOTL)
  • Abuso de credenciais válidas (T1078)
  • Movimento lateral via protocolos nativos
  • Implantes customizados de longa duração
  • Exfiltração lenta e fracionada
  • Foco em persistência silenciosa vs. impacto imediato
TTPs Principais (MITRE)
  • T1078 Credenciais Válidas
  • T1018 Remote System Discovery
  • T1046 Network Service Scanning
  • T1053 Scheduled Task/Job
  • T1087 Account Discovery
  • T1059.001 PowerShell

Fontes: CISA Advisory AA24-038A (fev/2024) · Mandiant M-Trends 2024 · Microsoft MSTIC Salt Typhoon tracking · CrowdStrike Adversary Profile

⚠ Nota de Design

Este lab usa o Windows Server 2025, que deprecou RC4 por padrão e exige SMB signing. Isso bloqueia técnicas legadas como Kerberoasting clássico e Pass-the-Hash via SMB não assinado — documentado como achado, não como falha do exercício. O Salt Typhoon real adapta tooling ao ambiente; a simulação reflete isso.

Fase 01 / 06

Reconhecimento Interno

FASE 01 Remote System Discovery & Network Scanning T1018 T1046 Ruído: Baixo

O objetivo desta fase é mapear o ambiente sem gerar assinaturas de malware. O Salt Typhoon prefere ferramentas nativas do Windows para enumerar hosts, shares e controladores de domínio. O operador estabeleceu foothold inicial via RDP com credenciais válidas roubadas (simuladas como tloa\jsmith).

Comandos Executados

PowerShell — Descoberta de DCs e hosts
# Enumerar todos os DCs do domínio
nltest /dclist:tloa.local

# Listar hosts via DNS
nslookup -type=SRV _ldap._tcp.tloa.local

# Descoberta de hosts via ping sweep (LOTL)
1..254 | ForEach-Object {
    $ip = "192.168.204.$_"
    if (Test-Connection -ComputerName $ip -Count 1 -Quiet -TimeoutSeconds 1) {
        Write-Output "ALIVE: $ip"
    }
}

# Enumerar shares em hosts descobertos
net view \\192.168.204.10 /all
Get-SmbShare -CimSession DC01
Port Scan via PowerShell (sem Nmap)
# Scan de portas críticas — nativo, sem ferramenta externa
$targets = @("192.168.204.10", "192.168.204.11")
$ports   = @(445, 389, 3389, 5985, 88)

foreach ($t in $targets) {
    foreach ($p in $ports) {
        $conn = New-Object System.Net.Sockets.TcpClient
        $async = $conn.BeginConnect($t, $p, $null, $null)
        $wait  = $async.AsyncWaitHandle.WaitOne(300, $false)
        if ($wait) { Write-Output "OPEN: $t`:$p" }
        $conn.Close()
    }
}

Eventos Gerados e Análise de Detecção

Evento Origem Descrição Wazuh Alert
Sysmon EID 1 WS01 Criação de processo: nltest.exe /dclist ✓ Alerta
Sysmon EID 3 WS01 Conexão TCP em rajada para porta 445/389 ✗ Silêncio
WinEvent 4688 WS01 PowerShell.exe spawnou processo filho △ Parcial

Gap de Detecção

O scan de portas via TcpClient nativo gerou zero alertas no Wazuh. A telemetria do Sysmon (EID 3) registrou as conexões, mas a regra padrão não correlaciona volume de conexões TCP por processo em curto período. Um threshold customizado é necessário: ≥10 conexões TCP de um único processo em <5s para hosts distintos.

Fase 02 / 06

Living Off the Land (LOTL)

FASE 02 Inventário do Ambiente via Ferramentas Nativas T1059.001 T1082 Ruído: Mínimo

O Salt Typhoon evita dropping de binários externos ao máximo. Toda coleta de informações é feita com ferramentas já presentes no Windows. Esta fase representa o coração da dificuldade defensiva: o comportamento é idêntico ao de um administrador legítimo.

Comandos Executados

PowerShell — Inventário de ambiente
# Informações do sistema e AD
Get-ADDomain                          # estrutura do domínio
Get-ADForest                          # florestas e trusts
Get-ADTrust -Filter *               # relações de confiança
Get-ADComputer -Filter * -Properties OperatingSystem, LastLogonDate

# GPOs aplicadas — mapa de controles de segurança
Get-GPO -All | Select-Object DisplayName, GpoStatus, ModificationTime

# Serviços em execução — alvos para persistência
Get-Service | Where-Object { $_.Status -eq 'Running' } | Select-Object Name, DisplayName

# Processos com path de executável — detecção de EDR
Get-Process | Select-Object Name, Id, Path | Where-Object { $_.Path -ne $null }

# Usuários ativos por workstation — padrão de uso
query user /server:WS01

Por Que Isso é Difícil de Detectar

ComportamentoLegítimo?Distinguível?

Get-GPO -All por conta não-adminLeitura permitida por padrão✗ Não sem baselining

Gap de Detecção

Todos os comandos LOTL desta fase passaram sem nenhum alerta no Wazuh com configuração padrão. O Sysmon capturou os processos (EID 1), mas sem regras de correlação para "sequência de enumeração AD em menos de 10 minutos por conta não-privilegiada", o volume de eventos é tratado como ruído normal de ambiente.

Fase 03 / 06

Abuso de Credenciais Válidas

FASE 03 Valid Accounts — Logon com Credenciais Comprometidas T1078 Ruído: Médio

Esta é a TTP central do Salt Typhoon. Uma vez obtidas credenciais válidas (via phishing, credential stuffing ou compra em fóruns), o ator autentica-se como se fosse o usuário legítimo. O desafio defensivo é distinguir o acesso malicioso do legítimo quando as credenciais são as mesmas.

Cenário Simulado

Credenciais de tloa\svc_backup (service account) foram "obtidas" fora do horário comercial. O ator autentica via RDP para DC01 às 02:17 de uma workstation que normalmente não usa essa conta.

O Wazuh gerou alerta parcial no EID 4624 (regra genérica de logon fora de horário), mas não correlacionou os três eventos (4624 + 4648 + 4672) como uma cadeia suspeita. Fase 04 / 06

Persistência via Scheduled Task

FASE 04 Scheduled Task/Job — Persistência Silenciosa T1053.005 Ruído: Médio-Baixo

O Salt Typhoon abusa de scheduled tasks para garantir reexecução de payloads após reboot ou reconnection. A técnica é especialmente eficaz porque tarefas agendadas são operação rotineira em ambientes Windows — o sinal se perde no ruído.

Simulação

CMD — Criação de Scheduled Task camuflada
# Tarefa com nome que imita manutenção legítima
schtasks /Create
  /SC DAILY
  /TN "\Microsoft\Windows\Maintenance\SyncDiagnostics"
  /TR "powershell.exe -NonInteractive -WindowStyle Hidden -EncodedCommand [BASE64]"
  /ST 03:00
  /RU SYSTEM
  /F

# Verificação se tarefa foi criada
schtasks /Query /TN "\Microsoft\Windows\Maintenance\SyncDiagnostics" /FO LIST
Windows Event Log — EID 4698, 4702

  • Task criada em namespace \Microsoft\Windows\Maintenance\ por conta não-SYSTEM

Fase 05 / 06

Coleta e Enumeração de Contas

FASE 05 Account Discovery — Mapeamento de Alvos de Alto Valor T1087.002 Ruído: Baixo

Antes de qualquer movimento lateral, o ator mapeia contas de alto valor: membros de grupos privilegiados, service accounts com permissões elevadas, e contas com SPN (alvos de Kerberoasting). O objetivo é identificar o próximo passo de escalação.

PowerShell — Enumeração de grupos privilegiados
# Membros do Domain Admins
Get-ADGroupMember -Identity "Domain Admins" -Recursive

# Enterprise Admins — controle de floresta
Get-ADGroupMember -Identity "Enterprise Admins" -Recursive

# Service accounts com SPN — alvos de Kerberoasting
Get-ADUser -Filter { ServicePrincipalName -like "*" } -Properties ServicePrincipalName, PasswordLastSet, LastLogonDate

# Contas com PasswordNeverExpires — alta probabilidade de senha fraca antiga
Get-ADUser -Filter { PasswordNeverExpires -eq $true } -Properties PasswordLastSet

# AdminSDHolder — contas protegidas (alto valor)
Get-ADUser -LDAPFilter "(adminCount=1)" -Properties adminCount, MemberOf

Gap de Detecção

Toda esta enumeração via Get-ADUser e Get-ADGroupMember gerou zero alertas. São operações LDAP normais, autorizadas para qualquer usuário autenticado por padrão no AD. A detecção requer baselining de volume: quantas queries LDAP um usuário normalmente faz por hora? Uma explosão de 50+ queries em 3 minutos é anomalia comportamental, não violação de regra.

Fase 06 / 06

Threat Hunting — Fechando o Loop

Com as fases anteriores documentadas, o hunting parte de hipóteses concretas derivadas do comportamento simulado. Cada hipótese tem uma query testável.

Hipótese H-01
Conta de serviço autenticou via RDP em host diferente do esperado em horário off-hours
KQL — Wazuh / Elastic
event.code:"4624"
AND winlog.event_data.LogonType:"10"
AND winlog.event_data.TargetUserName:"svc_*"
AND NOT winlog.event_data.WorkstationName:("WS01" OR "WS02")
AND @timestamp:["00:00" TO "06:00"]
Hipótese H-02
PowerShell com -EncodedCommand criou scheduled task em namespace do sistema
Sigma Rule (simplificada)
title: Scheduled Task Created With Encoded PowerShell
logsource:
  product: windows
  service: security
detection:
  selection:
    EventID: 4698
    TaskContent|contains|all:
      - 'EncodedCommand'
      - 'Hidden'
      - 'NonInteractive'
  filter_legit:
    SubjectUserName|startswith: 'svc_sched'   # conta autorizada
  condition: selection and not filter_legit
level: high
Hipótese H-03
Explosão de queries LDAP por conta não-administrativa em curto intervalo
KQL — Correlação temporal
event.code:"4662"                    // Object Access no AD
AND winlog.event_data.ObjectType:"user"
AND NOT user.name:("administrador" OR "svc_ad_mgmt")
// Agrupar por user.name, contar em janela de 5 minutos
// Threshold: count > 20 → alerta
| stats count() by user.name, date_histogram(field="@timestamp", fixed_interval="5m")
| where count > 20
Hipótese H-04
nltest.exe ou net.exe executado por conta não-admin fora do horário de helpdesk
KQL — Process Creation
event.code:"1"                       // Sysmon Process Create
AND process.name:("nltest.exe" OR "net.exe" OR "net1.exe")
AND process.command_line:("*dclist*" OR "*domain_trusts*" OR "*view*")
AND NOT user.name:("administrador" OR "svc_monitoring")
AND @timestamp:["18:00" TO "08:00"]
Conclusão

Binários externos dropados pelo "ator"

O Salt Typhoon demonstra que a fronteira entre administrador legítimo e ator malicioso não está nos comandos executados, mas no contexto em que são executados. Quatro das seis fases simuladas passaram sem nenhum alerta no Wazuh com configuração padrão — não por falha da ferramenta, mas porque as ações eram tecnicamente indistinguíveis do comportamento normal.

Os achados centrais são três: primeiro, o Wazuh captura a telemetria correta via Sysmon, mas falta correlação temporal e comportamental entre eventos; segundo, o Windows Server 2025 bloqueou ativamente técnicas legadas (RC4, SMB unsigned), forçando o ator simulado a adaptar tooling — isso é uma postura defensiva real, não um obstáculo ao lab; terceiro, a detecção eficaz contra LOTL exige baselining comportamental, não apenas regras baseadas em assinatura.

Recomendações para o Ambiente
  • Habilitar auditoria de LDAP Query Logging no DC (Event 1644) para visibilidade de enumeração AD
  • Criar regras de correlação Wazuh para cadeia 4624+4648+4672 por service accounts
  • Implementar baselining de uso de PowerShell por conta (volume de comandos/hora)
  • Alertar em qualquer EID 4698 com EncodedCommand no TaskContent
  • Restringir Get-ADGroupMember sobre grupos sensíveis via ACL no AD
